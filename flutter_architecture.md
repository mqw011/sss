# Flutter Architecture Guidelines (Project "999")

## 1. Архитектурная парадигма
Используется **Feature-first** (по-фичевой) подход, основанный на принципах **Clean Architecture**. 
Стейт-менеджмент: **Riverpod**.

## 2. Структура директорий (`lib/`)
```text
lib/
├── core/                       # Общие модули для всего приложения
│   ├── theme/                  # Цвета, шрифты (строго по ui_brand_book.md)
│   ├── widgets/                # Переиспользуемые UI компоненты (PolaroidCard, GlassAppBar)
│   └── utils/                  # Хелперы, константы, расширения
├── features/                   # Модули приложения, разбитые по фичам
│   ├── genres/                 # Глобальные жанры (Уровень 1)
│   ├── subgenres/              # Узкие сообщества / Поджанры (Уровень 2)
│   └── chat/                   # Обсуждения (Уровень 3)
└── main.dart                   # Точка входа, конфигурация ProviderScope
```

## 3. Внутренняя структура фичи (Pattern)
Каждая фича внутри `lib/features/<feature_name>/` делится на слои:
*   `domain/`
    *   `entities/` — чистые модели данных (например, `Message`, `Genre`).
    *   `repositories/` — абстрактные интерфейсы (контракты, например, `IChatRepository`).
*   `data/`
    *   `repositories/` — реализация абстракций (например, `MockChatRepository` для начала).
    *   `dto/` — классы для парсинга (опционально).
*   `presentation/`
    *   `screens/` — UI экраны фичи (управляют навигацией, слушают стейт).
    *   `widgets/` — локальные виджеты (специфичные для экрана).
    *   `providers/` — Riverpod провайдеры (Связывают UI с Domain/Data).

## 4. Золотые правила кода
1. **Separation of Concerns:** UI-слой ничего не знает о том, откуда берутся данные. Он слушает провайдеры Riverpod.
2. **Dependency Inversion:** Провайдеры общаются с Data-слоем исключительно через абстрактные интерфейсы (`IChatRepository`). Это позволит в будущем заменить мок-репозиторий на Firebase одной строчкой конфигурации.
3. **Immutability:** Модели (Entities) и классы состояний (State) должны быть иммутабельными.
